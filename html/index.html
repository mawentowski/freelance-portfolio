API Conceptual docs

SITA Flex APIs

The Company: SITA

SITA is one of the world's leading specialist in air transport communications and information technology companies that has products covering all major operations at airports.

The Deliverable: API Conceptual Docs

A Software Development Manager at SITA approached me about making significant updates to the SITA Flex APIs Developer Guide for an upcoming V2 release. He was spending hours a week on support calls with customers for issues that otherwise could have been solved with full-written documentation.

The Flex Developer Guide is "conceptual" documentation designed to build an intuition the developer's mind about what SITA Flex is and how it can be used. It contained sections likes the APIs product overview, getting started, authentication, tutorials, code samples, and status/error codes.

The platform hosted 6 separate but related APIs, each with their own complex business processes that needed to be documented alongside the technical content. In addition, the APIs required developers to use the Azure Service Bus to receive device messages. So, not only did I need to master using Flex APIs, but I also needed to understand how the Azure Service Bus integrated with Flex in an intimate way.

Over the period of 2 months, we are able successfully publish the conceptual documentation alongside the reference docs on the Dev Portal. According to the Software Development Manager, support requests were cut down significantly and continued to decrease as the docs were refined in subsequent releases based on user feedback.

SITA Airport Operations API

API Reference docs

SITA Airport Operations API

The Company: SITA

SITA is one of the world's leading specialist in air transport communications and information technology companies that has products covering all major operations at airports.

The Deliverable: API Reference Docs

I had the opportunity to take part in a dev team developing an API from the ground-up using the "spec-first" philosophy that involved writing an Open API spec by hand prior to developing the API. The team adopted this approach after delivering an API that severely lacked user input when it came to the endpoints and parameters so there were many things the users felt they could not accomplish in the way they would like to. The goal was to make the Open API Spec a contract for developers to build their API against.

This portfolio item is based on the hand-written spec and demonstrates my knowledge of the Open API spec, facility with its concepts, and how to render that data in a branded documentation website.

Because the spec was being written from the ground up, I had a chance to be involved in the user research. I was on all calls with the customer to help build personas based on their unique needs for the different personas.

I actively participated in the creation of the Spec. This effort also got me involved in taking an active part in adding descriptions to the endpoints and parameters in the Spec. Defining the endpoints and chronology of how the APIs were used, I was able to contribute to the user experience of the API. I was involved in the ground up to help write these descriptions as opposed to later.

After the developers had developed the API to match the Open API Spec, they then could render the spec using a tool like Swagger.
